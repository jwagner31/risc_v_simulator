Introduction

In this part, we are going to make some changes to MIPS R4000 pipeline. The MIPS R4000 family processors implement MIPS64. They use deeper pipeline than the simple 5-stage design. This deeper pipeline allows it to achieve higher clock rates by decomposing the five-stage integer pipeline into 8 stages. Since the cache access stages are usually on the critical path, the extra pipeline stages come from decomposing the memory access stages to achieve more balanced pipeline stages. Appendix C in the book is a good resource for learning classical 5-stage pipelining. You can also refer to chapter 3 in the MIPS R4000 user's manual which explains the MIPS R4000 family pipeline.

Description of pipeline

Our pipeline has eight stages. Although the instruction and data memory occupy multiple cycles, they are fully pipelined, so that a new instruction can start on every clock. The function of each stage is given as follows: (NOTE: The stages described below are different from those of MIPS 4000.)

  • IF - First half of instruction fetch. PC (Program Counter) selection actually happens here, together with initiation of instruction cache access.
  • IS - Second half of instruction fetch, complete instruction cache access. Note that in this project we assume instruction accesses always hit the instruction cache.
  • ID - Instruction decode, hazard checking.
  • RF - Register fetch.
  • EX - Execution, which includes effective address calculation, ALU operation, and branch-target computation and condition evaluation.
  • DF - Data fetch, first half of data cache access.
  • DS - Second half of data fetch, completion of data cache access. Note that the data access always hit the data cache.
  • WB - Write back for loads and register-register operations.

Your program is expected to detect various hazards, stalls and forwarding needed in the pipeline and perform correct operations in the simulator. Logically, all the detections are performed at ID stage.  Note that Figure C.24 is a good example for harzards and forwarding detections but the table used in this project might not be exact same with the table in C.23.

Load hazard stalls: In addition to substantially increasing the amount of forwarding required, this longer-latency pipeline increases both the load and branch delays. This is encountered when RAW register dependence results from a memory load instruction. In this case, the new register value is not known until after the DS stage since it must be loaded from memory. If an instruction is immediately followed the load and needs the value in the EX stage, two stall cycles must be inserted.

Branch hazard stalls: Since branches are resolved during EX, the basic branch delay is 4 cycles. In this project assume that there is no delayed branch. We use a predicted-not-taken strategy for reducing the branch delay. Therefore, for untaken branches, there will be no delay. However, if the outcome is taken, the instruction in the IF, IS, ID and RF stages must be cancelled (four bubbles) and the target instruction will be fetched at the next cycle. For unconditional jumps, four stall cycles are always created.

Forwarding: In the 8-stages pipeline, there are many opportunities for data forwarding.

  • EX/DF => RF/EX
  • DF/DS => EX/DF
  • DF/DS => RF/EX
  • DS/WB => EX/DF
  • DS/WB => RF/EX

Assumptions

  • As in part I, assume the program starts at memory location 496 (decimal). PC is initialized to this location for fetching the first instruction out of the memory. The data region starts at location 600 unless the instruction region exceeds location 600. In such cases, the data immediately follow the BREAK. Assume further that your program and data are allocated within the physical memory region from 496 up to 1000 (decimal)
  • Assume the effective address is the same as the physical memory address.
  • Instruction issue is static and in-order, just like the 5-stage pipeline.
  • Proper pipeline registers must be used to latch intermediate results between pipeline stages.
  • A multiple-ported register file is assumed to allow concurrent reads and writes to registers, i.e. there will be no resource hazard for register reads and writes.
  • In your simulator, there is no delay branch slot.
  • In this part of the project, you need not worry about multi-cycle execution instructions.

Guidelines

  • Your output should match the following formats as closely as possible.
  • Your simulated pipeline will be initialized to contain all NOP instructions.
  • Your simulator should simulate the actual execution and produce the correct results for the given program.
  • After a "BREAK" instruction is decoded, your simulator should complete all the instructions before the BREAK.
  • You should buffer the pipeline with "NOP" instructions to indicate empty stages after the BREAK.
  • A program will be considered "complete" once the BREAK instruction leaves the WB stage.
  • A stall should be represented as "** STALL **" in showing your pipeline
  • Please be careful that your cycle-based output reflects the machine status after the given cycle has finished.
  • "Stall Instruction" refers to the instruction that was stalled in issue and not advanced to the next stage during the cycle.
  • Print out the decoded instruction that is stalled after "Stall Instruction:", or "(none)" if no instruction was stalled this cycle.
  • You will report forwarding as it occurs by printing the register name (e.g., "R3") that refers to the value being forwarded next to the appropriate forwarding data line.
  • Please note that the register name does not necessarily contain the value being forwarded; rather, you are pointing out the hazard which is being resolved by associating the register name with the stages in the pipeline where the forwarding is occurring.
  • "Total Stalls" and "Total Forwardings" refer to the total number of stalls inserted into the pipeline and the total number of forwardings that have occurred up until and including the current cycle.

Output

  • Following command line options will be supported: "-T" - Trace mode - ( See the example for format )
  • Refer to the main project page for information on how this is specified when running the program.
  • If Tm:n is specified, you are to output tracing information for each cycle m through n, inclusive. Your simulator should still execute the entire program and print out a final summary, but only print tracing information for the specified cycles. You may refer to Figure A.19 for pipeline register notations.
  • If T0:0 is specified, simulate the entire program and output only the summary information. Do not print out any tracing information.
  • If Tm:n is not specified, output tracing for the entire program.
  • After execution, your program should output the following summary information. (See the attached example for details.)
    • All the register and memory contents. Output the signed integer values (32 bits) for the registers and the data portion of memory.
    • Total number of simulated cycles.
    • The number of load stalls, branch stalls, and other stalls.
    • The number of forwarding for each type.
